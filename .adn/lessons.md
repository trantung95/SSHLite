## 2026-06-16 - Making a positional-composite-key field optional is a whole-codebase change, not a type tweak

**Context**: design to make `IHostConfig.username` optional (an "endpoint" host = `host:port` with no account yet, accounts added later via "Add User"). A 5-agent full-source audit found the change ripples far past `types.ts`. Reusable gotchas for ANY time a field that participates in a composite id/key (`a:b:c`) becomes optional or gains an empty/sentinel value:

- **`${undefined}` interpolates to the literal text `"undefined"`, not an empty segment.** ids built as `` `${host}:${port}:${username}` `` become `"host:port:undefined"` — poisoning credential keys, tab prefixes (`connectionPrefix.ts`), and import conflict detection. Fix: normalise the optional field to `''` and centralise build/parse in ONE helper (`buildHostId`/`parseHostId`) that splits from the RIGHT (last segment may be empty; first segment may itself contain `:` for IPv6). This also fixes the pre-existing IPv6 `split(':')` corruption that was already latent at `HostService` removeHost/renameHost/setTabLabel, `FileDecorationProvider`, `PortForwardTreeProvider`.
- **Empty string is falsy, so existing `!x` guards silently DROP the record.** `if (!h.username) continue;` at load/export/import quietly deletes every endpoint with no error. Audit every truthy guard on the field and make it intent-aware (`!h.username && !h.isEndpoint`). Add an explicit boolean flag rather than detecting "emptiness" — the flag is self-documenting and survives round-trips.
- **`.toLowerCase()` / string ops on the now-optional field crash at runtime** (`HostTreeProvider.matchesFilter`). TypeScript only catches these once the type is actually `string | undefined`; until then they are landmines.
- **Collapse the blast radius with ONE chokepoint.** Most cosmetic `undefined@host` bugs lived on the "connected" path. A single guard that makes an endpoint un-connectable (at `ConnectionManager`/`ConnectionFactory`, plus the empty-username guard `SSHConnection` already had but `FTPConnection` did NOT — FTP `client.access({user:''})` silently logs in anonymous) makes all of them unreachable. Find the one boundary that gates the whole class before fixing N call sites.
- **VS Code settings schema** (`package.json contributes.configuration`) independently declares the field required — an optional runtime value flags the whole settings array invalid in the Settings UI. Update the schema too.
- **Export/import is the highest-risk surface**: types (`ExportedHost`), per-entry validation, envelope version + a `containsEndpoints` flag for forward-compat, and any module with its OWN local id builder (`connectionSyncCommands.ts` had a private `hostId()`) must all be updated or conflict detection silently breaks. Full checklist: `docs/specs/2026-06-16-connect-time-accounts-audit.md`.

## 2026-06-14 - Running the FTP docker suites: containers are torn down between separate jest runs

**Gotcha**: `jest.docker.config.js` shares one `globalSetup`/`globalTeardown` for ALL docker tests (SSH and FTP). The teardown runs `docker compose -f test-docker/docker-compose.yml down`, which removes **every** service in that compose file - including the manually-started `ftp` (vsftpd, 2207) and `ftp-pure` (pure-ftpd, 2208) containers. So each separate `npx jest --config jest.docker.config.js -- docker-ftp` invocation leaves the FTP containers gone; the NEXT FTP run must `docker compose -f test-docker/docker-compose.yml up -d ftp ftp-pure` again and re-pay the pure-ftpd cold-boot warmup (30-90s of cert + DH-param generation, absorbed by `connectWithRetry` in `ftpTestHelpers.ts`). The SSH containers (2201-2204) auto-start via `globalSetup`, but the FTP ones do not - only the teardown touches them. **Rule when re-running FTP tests N times**: restart `ftp ftp-pure` before each run, or run all FTP suites in ONE jest invocation (`-- docker-ftp` matches all four) so the single teardown happens once at the end.

**Verification result (issue #9 FTP feature)**: full audit found NO bugs and backward-compat SAFE. compile 0 errors; unit 1863 pass (x2, stable); docker-ftp 71 pass (real vsftpd + pure-ftpd + explicit FTPS); docker-ssh 140 pass (proves the shared `IConnection` refactor did not regress SSH at runtime); stress+fileservice re-run 52 pass (serialization queue is not flaky). Backward compat rests on: optional `connectionType` defaults to `'ssh'` (`getConnectionType` in HostService), SSH `contextValue`s unchanged with FTP getting a `.ftp` suffix, SSH-only menu `when` clauses using `(?!\.ftp)` negative-lookahead, and `capabilityGuard` (`ensureCapability`/`assertCapability`) backstopping every SSH-only service sink so an SSH-only call on an FTP connection warns/throws instead of a TypeError.

## 2026-06-11 - Native remote search tools (ripgrep/fd/parallel-grep/plocate) with per-server auto-detection + 3-tier fallback

**Feature**: remote search/filter now detects and uses faster OS-native tools per connection instead of hardcoding `grep -rnHI` / `find`. See `.adn/features/search-system.md` ("Native tool selection"). Pure command construction lives in `src/connection/searchCommandBuilder.ts` (no ssh2/vscode — fully unit-testable); `SSHConnection.searchFiles()` probes once per connection, selects a strategy, executes, and falls back. Reusable lessons:

- **busybox `grep` silently returns 0 results with `--include`/`--exclude-dir`.** busybox grep does NOT support those GNU flags: it exits 2 and prints to stderr, but the legacy command's `2>/dev/null | head` swallowed both → a content search WITH a file filter returned an empty result set and looked like "no matches" (a latent LITE "true data, no missing" violation that the docker images masked because `Dockerfile.sshd`/Alpine all `apk add grep findutils coreutils`). **Rule**: when shelling out with GNU-only flags, detect the tool flavor (`grep --version` → "GNU grep") and use a portable construction for non-GNU (`find -print0 | xargs -0 grep -nH`); add a busybox docker image (no GNU userland) so the gap can't hide. Never let `2>/dev/null` hide a tool-incompatibility that yields wrong (fewer) results.

- **ripgrep needs `--no-ignore --hidden` to match `grep -r` semantics.** `grep -r` respects no ignore files and walks hidden files; rg by default skips `.gitignore`d and hidden files. Swapping grep→rg WITHOUT those two flags silently returns FEWER results. The flag presence is asserted by an un-deletable unit test with a comment explaining why. rg may return a SUPERSET (it matches UTF-16-BOM files grep -I skips) — that is allowed under "true data" (more true results, never fewer) and documented.

- **rg can emit non-`file:line:text` lines** ("path: binary file matches ...") that grep -I silently skips. The content parser drops lines whose second colon-field is not numeric ONLY on the rg path (`requireLineNumber` flag on `BuiltSearchCommand`); the legacy grep path keeps its exact behavior.

- **`LC_ALL=C` speeds grep but can drop matches — guard it.** Only safe when NOT regex, query is pure ASCII, AND (case-sensitive OR no letters). GNU grep `-i` in a UTF-8 locale Unicode-case-folds (U+212A KELVIN matches `k`); C locale would silently miss those lines. `shouldUseCLocale()` encodes the predicate; case-sensitive ASCII fixed-string is provably byte-identical (UTF-8 self-synchronizing).

- **find `! -path '*/dir/*'` still DESCENDS the excluded dir, then filters.** Switching to `\( -name 'dir' \) -prune -o <match> -print` stops descent entirely (big win on node_modules/.git) and is busybox-safe. Explicit `-print` is mandatory once `-prune` is present, else the pruned branch also prints. Equivalent-or-identical result set to the old excludes.

- **Three-tier fallback so a native tool can never make search worse than grep/find:** (1) detection — missing tool/probe failure → legacy from the start; (2) execution — native commands omit `2>/dev/null` so stderr is visible; `shouldFallbackToLegacy({resultCount, stderrText, aborted, execError})` re-runs the legacy command ONCE when the native one errored or returned 0-with-stderr (the silent-exit-2 class), but NOT on a clean 0 (genuinely no matches — re-running would double server load, LITE) nor on user-abort; (3) memory — after a fallback the tool is marked degraded on the connection's `RemoteSearchTools` cache so later searches skip it. Reconnect re-probes (a fresh `SSHConnection` is created per connect).

- **Detection is lazy + user-triggered (LITE).** The probe (`buildToolProbeCommand()` — one POSIX/busybox-safe exec returning tool paths + grep/xargs flavor + `uname -s` + `nproc`) runs INSIDE the first `nativeTools:'auto'` search, memoized via `_remoteToolsPromise` (same pattern as `_sftpPromise`), never at connect time. `parseToolProbeOutput()` never throws on garbage/partial output and requires absolute paths (rejects shell-builtin names from `command -v`).

- **Refactor proof = exact-string tests stay green.** The byte-identical-grep unit tests (`SSHConnection.test.ts` "searchFiles") were the contract for extracting the builders; only the find tests changed (for `-prune`). Setting default: `sshLite.searchNativeTools` defaults to `'auto'`, but the `searchFiles` option defaults to `'off'` so any caller that doesn't pass it (and every existing unit test) stays on the legacy path with no probe — callers (`SearchPanel`, `FileTreeProvider`) read the setting and pass it explicitly.

- **Indexed-search gotchas (opt-in plocate + client snapshot, found in code review):**
  - **A prefix `grep -F -- '<basePath>'` must carry a trailing slash.** `grep -F -- '/home/alice'` ALSO matches `/home/alice2/...`; with a `| head -N` after it, those sibling false-positives eat the budget and silently drop real `/home/alice/...` results (LITE no-missing break) — even though a later client-side `startsWith('/home/alice/')` filter would have removed them, the data was already lost at `head`. Anchor BOTH the server prune and the client filter with `basePath + '/'`. (`buildLocateCommand`.)
  - **Distinguish user-abort from a real refusal in a result code.** A long index build returned `{refused:'too-large'}` on `signal.aborted`, so cancelling showed a bogus "folder too large" warning. Give abort its own `'aborted'` code and have the command no-op silently on it. Any cancellable long task that returns a status union needs a dedicated cancelled state, distinct from failure states.
  - **`stat -c %y` is unparseable by `Date.parse` (nanoseconds + ` +0000`); use `%Y`** (integer epoch seconds) for any mtime you'll compute an age from.
  - **A filename index must REFUSE rather than truncate.** `buildIndex` passes `maxResults = cap` and treats `rows.length >= cap` as "possibly truncated → refuse" — never store a partial filename index (it would silently miss files). Surface the cap to the user.
  - **Client snapshot key must be host-stable, not connection-id.** `connectionId` changes every reconnect; key snapshots by `host:port:user::basePath` so they survive reconnects. (`FilenameIndexService.hostKey`.)

## 2026-06-10 - Connection Import/Export/Sync + Google Drive OAuth (issue #11): secret-free exports, drive.file scope, loopback+PKCE, async-return-unwrap

**Feature**: see `.adn/features/connection-portability.md` for the full design. Key reusable lessons:

- **Export what the user SEES, not just what you store.** The SSH Hosts panel shows `getAllHosts()` = `~/.ssh/config` hosts + saved hosts (merged, deduped). The first cut of export read saved-only (`sshLite.hosts`), so a real user with 19 saved + 82 ssh-config hosts got an export of 19 and a "19 connections" popup — silent data loss + a count that didn't match the panel (LITE "true data, no loss" violation). **Rule**: any export/backup/sync of a list the user perceives must be sourced from the SAME function that builds the panel/visible view, not from one underlying store. Here: `getAllHostsForExport()` returns the full deduped `getAllHosts()` set. Confirm the count by reading the user's actual `sshLite.hosts` (in `%APPDATA%/Code/User/settings.json`) AND `~/.ssh/config` Host entries — don't trust a single source.
- **Export the unexpanded path, not the resolved one.** `getAllHosts()`/`loadSSHConfigHosts()` run `expandPath()` (`~/.ssh/id_rsa` -> `/home/alice/.ssh/id_rsa`), machine/OS-specific and broken on another box. For portability: saved hosts read the RAW stored value (`getSavedHostsForExport()` reads `sshLite.hosts` directly so `~` survives); ssh-config hosts get their expanded path **collapsed back to `~`** (`expandPath` is just `p.replace('~', os.homedir())`, so reverse it with `startsWith(home)`). On import all hosts become `source: 'saved'`; dedup-by-id keeps the panel clean on the same machine and restores hosts whose `~/.ssh/config` is absent on a new machine.
- **Secrets never leave SecretStorage.** Exports carry only credential metadata (label/type/key-path) + pinned folders; an imported password credential has no stored secret, so `SSHConnection.buildAuthConfig` prompts on first connect (proven in `docker-ssh-import.test.ts`). The "no secret" guarantee is enforced two ways: a whitelist in `CredentialService.exportMetadata()` (strips any planted field) AND a `CredentialType` allowlist on import (`importCredentialMetadata` drops creds whose `type` isn't `password`/`privateKey`, so a crafted file can't pollute the credential index). Regression: `ConnectionPortabilityService.test.ts` plants a secret and asserts it is stripped.
- **Google Drive from an extension = loopback + PKCE (S256), Desktop client, `drive.file` scope.** `drive.file` and `drive.appdata` are NON-sensitive scopes — they avoid Google's costly CASA security assessment that the broad `drive`/`drive.readonly` scopes trigger. Use `drive.file` for a user-visible synced file. A Desktop client's "client secret" is NOT confidential (it ships in the extension); PKCE is the real protection, so shipping it is fine. While the OAuth consent screen is in "Testing", Google expires refresh tokens after 7 days — must publish to "In production". Do it all with raw `fetch` (Node 18+/global) — do NOT add `googleapis` (~30 MB). Upload host is `/upload/drive/v3/...`; metadata/list/download is `/drive/v3/...`; download needs `alt=media`. Tokens go in `vscode.SecretStorage`, refresh-then-retry once on HTTP 401.
- **`return somePromise` inside an `async` function is auto-unwrapped** — the async wrapper adopts the inner promise, so a caller's `await fn()` gets the resolved value, NOT a (truthy) Promise. So `async ensureSignedIn() { ...; return drive.isSignedIn(); }` is correct even without an inner `await`. Don't "fix" a missing `await` on a returned promise inside an async function as if it were a truthy-Promise bug — write a test first; it'll pass. (Add `return await` only for clearer stack traces, not correctness.)

## 2026-06-10 - Error handlers that fire a refresh must record the failure FIRST (issue #13); keybindings pass no tree-item args (issue #10)

**Issue #13 (infinite loop, frozen UI, notification spam)**: `loadDirectoryAndRefresh()`'s `.catch()` deleted the `loadingItems` key and then fired `_onDidChangeTreeData`. The refresh re-entered `getChildren()`, which saw "not cached, not loading" and started the load again → fail → notify → refresh → load, forever. **Rule**: in any retry-able async flow where the FAILURE handler itself triggers the code path that starts the work (here: tree refresh → `getChildren`), the handler must persist a "this failed, do not auto-retry" marker BEFORE triggering it, and the entry point must check that marker. Clearing the in-flight flag alone re-arms the loop. Retry must be tied to an explicit user action (refresh button, navigation, reconnect) — which also matches LITE (user-triggered only). Implemented as `failedLoads` map + `LoadErrorTreeItem` in `FileTreeProvider`; regression: `FileTreeProvider.issue13.test.ts`.

**Issue #10 follow-up ("SSH clipboard is empty")**: VS Code passes tree-item arguments to a command ONLY for context-menu invocations (`(item)`, or `(item, items[])` with multi-select). A **keybinding** invocation passes NO arguments, so `copy/cut/rename/paste` handlers guarded by `if (!item) return;` silently did nothing — the v0.9.11 fix made the hotkeys FIRE but they still no-oped, and paste then reported an empty SSH clipboard. **Rule**: every tree-view command that has a keybinding must fall back to `treeView.selection` (the `TreeView` instance from `createTreeView()` — keep it in scope) when its args are undefined, and should give feedback (status bar) instead of returning silently. Shared helper: `resolveTreeSelection()` in `src/utils/treeSelection.ts` (priority: `items[]` > `item` > selection). Audit the OTHER commands when adding a new keybinding — this class of bug recurs per keybinding added.

**Process note**: both bugs were reported by the same user minutes after a release. Hotkey/keybinding fixes need testing via the actual keybinding path (no args), not just the context menu path — unit-test the pure fallback logic and manually drive the hotkey.

**Docker coverage (issue #13 + #12)**: the SFTP-touching fixes got docker integration tests, not just mocks. `docker-ssh-listing-failure.test.ts` seeds a real `chmod 000` dir and proves (a) `listFiles()` actually REJECTS on it (the whole #13 fix depends on that — if it returned `[]` the loop fix would be untested), (b) the real `FileTreeProvider` renders the error item and a `jest.spyOn(conn,'listFiles')` confirms zero re-listing across repeated `getChildren`, (c) the notification fires once, (d) recovery after `chmod 755` + refresh. `docker-ssh-image-open.test.ts` writes a real PNG and a 1.5MB binary, opens through the real `FileService`, and SHA256-compares the on-disk temp file to prove the FULL bytes survived the SFTP round-trip (no truncation) while `vscode.open` was called and `showTextDocument` was not. Both files MUST be added to the `testMatch` array in `jest.docker.config.js` or they silently never run. Wiring a real connection into a provider: `(ConnectionManager.getInstance() as any)._connections.set(conn.id, conn)` (template: `docker-ssh-reveal.test.ts`).

**Tooling gotcha — stale containers block `globalSetup`**: `globalSetup.ts` runs `docker compose up -d --build`, which fails with "container name already in use" if a previous manual run (or the multi-OS / windows-client stack) left fixed-name containers (`sshlite-manual`, `sshlite-test-sudo`, `sshlite-os-*`, `sshlite-test-server-*`) around. Clean slate before a docker run: `docker compose -f test-docker/docker-compose.yml down --remove-orphans`, then `docker rm -f <leftover names>`.

**Feedback (workflow)**: if a task needs docker (real-server tests) and Docker Desktop is not running, ASK the user to start it BEFORE attempting the run — do not silently skip the docker step. The user wants the real-server coverage and will start Docker on request.

## 2026-06-10 - Verify a fix against the real INTERACTION/DESIGN model, not just the code path (issue #10, design flaw caught by user)

**Context**: the issue #10 fix made copy/cut/paste/rename fall back to `fileTreeView.selection` when a keybinding passes no tree-item argument. Unit tests passed, the code was correct — but it does NOT fix the reporter's actual scenario. The reporter copies a file by **mouse single-click then Ctrl/Cmd+C**. In VS Code a `FileTreeItem` with `this.command = { command: 'sshLite.openFile' }` (FileTreeProvider.ts ~342) means a single click OPENS the file, and `openRemoteFile()` calls `showTextDocument(doc, { preview: false })` with no `preserveFocus`, so focus moves to the **editor**. The copy keybinding is gated `when: focusedView == sshLite.fileExplorer`, which is now false, so Ctrl+C never reaches the SSH command at all — and even if it did, the tree is no longer focused. The selection-fallback only helps the cases where focus STAYS in the tree: clicking a folder (no editor opens) or arrow-key navigation (preview opens but tree keeps focus).

**Lesson — when fixing a UI bug, model the real user interaction end to end before declaring the root cause, and check for DESIGN flaws, not just code bugs:**

1. **A passing unit test that calls the command with synthetic args proves the handler logic, NOT that the feature is reachable in the real UI.** The keybinding `when` context, focus ownership, and what a click/keypress actually does to focus are part of the bug. Trace: input event → does the `when` clause hold? → which view has focus? → does the command even fire? Only then the handler logic.
2. **Enumerate the input modalities** (mouse single-click, double-click, arrow-key navigation, context menu, command palette, keybinding) and check the fix under EACH. They differ in focus behavior. A fix that works for keyboard navigation can be dead for mouse click.
3. **Distinguish a code bug from a design flaw.** Here the real issue is an interaction-design conflict: "single-click opens + steals focus" vs "Ctrl+C needs the tree focused + an item selected." No amount of handler-side patching fixes that; it needs a design decision (e.g. open with `preserveFocus: true` so the tree keeps focus like VS Code's native Explorer preview, vs. accepting that keyboard-copy requires arrow-key selection or right-click). Surface the design tension to the user instead of shipping a code patch that can't reach the bug.
4. **Reproduce in the real product, not just tests, for any UI/UX bug** — the docker manual-test setup (real server + the actual extension build) would have exposed this immediately. Manual repro is part of "prove it works," especially for focus/keybinding/WebView behavior that mocks cannot capture.

**Resolution (v0.9.13)**: closed by giving `openRemoteFile()` a `preserveFocus` option and passing `preserveFocus: true` from the tree-click `openFile` command — the file opens (permanent tab unchanged) without stealing focus, so the tree keeps focus + selection and Ctrl/Cmd+C/X/F2 work right after a click (matches VS Code's native Explorer). Callers that should land in the editor (create-file, new-file-as-root, search go-to-line) omit the option and keep focus-the-editor. The selection-fallback from layer 1 still does the actual copy off `fileTreeView.selection`. Two layers, both needed: the command must (a) be reachable with the tree focused AND (b) find the item — fixing only one leaves the bug.

## 2026-06-07 - Changing code that backs a past fix: re-prove the old invariant (issue #8)

**Context**: GitHub issue #8 asked for compact editor tabs (drop the `[user@host]` prefix). The fix edits `buildLocalTempPath()` in `src/utils/connectionPrefix.ts` - the SAME function that carries the issue #6 collision-free-temp-path fix. The user's standing instruction: *when fixing a bug, do not let previously-fixed bugs recur; verify carefully.*

**Lesson**: Editing a function that implements a past fix is the highest-risk place for silent regressions. Before changing it:

1. **Find what it backs.** Grep the file and its tests for issue numbers / read the related `.adn/` doc. `connectionPrefix.ts` backs issue #6 (collision-free paths) and feeds issue #7 (reveal). The change must touch ONLY the new concern (the filename prefix) and leave the load-bearing part (the `connHash`/`dirHash` directory layout) untouched.
2. **Add a regression assertion for the OLD invariant under the NEW behavior.** The new test "keeps temp paths collision-free across folders regardless of prefix mode (issue #6 invariant)" proves dropping the prefix still gives same-named files in different folders distinct temp paths.
3. **Re-run the prior regression tests**, not just the new ones - unit (`connectionPrefix.test.ts`) and docker (`docker-ssh-collision.test.ts`, which calls `getLocalFilePath` in the default mode).

**Design note (issue #8 itself)**: SSH Lite opens a remote file as a local temp file, so the temp filename IS the editor tab title (no API to set a custom label for a plain text editor). The prefix is therefore baked into the filename. `setTabPrefixMode()` + `buildTabPrefix()` keep the pure path builder decoupled from vscode settings (extension.ts pushes the `sshLite.editorTabPrefix` value in on activation + on change, validated against the known modes), mirroring the existing `registerTabLabel()` pattern. The prefix is cosmetic only: save-to-upload keys on `normalizeLocalPath(localPath)` via `getFileMapping()` and the save listener fires on `isInSshTempDir()` (path-based), so no prefix change can misroute an upload.

**Latent bug the prefix-in-filename design hides (found + fixed during issue #8)**: because the prefix is part of the temp filename, `openRemoteFile`'s path-keyed "already open" check (`textDocuments.find(d => d.uri.fsPath === localPath)`) breaks the moment the prefix CHANGES while a file is open. Change `editorTabPrefix` (or a host `tabLabel`), re-open the same remote file, and the recomputed local path no longer matches the open tab -> a SECOND editable temp copy of one remote file opens. Edit both, save both, and they overwrite each other on upload (silent lost edit). This was already reachable pre-issue-#8 through the `tabLabel` command. Fix: `findOpenLocalPathForRemote(connectionId, remotePath)` scans `fileMappings` for a still-open doc matching the remote file by connection+path (NOT filename) and `openRemoteFile` focuses it first. **General rule**: if a temp filename encodes mutable display state, never use that filename as the identity key for "is this already open / is this the same resource" - key on the stable identity (connection + remote path) instead. This bug lives only in in-memory mapping + editor state (no SSH/SFTP/event-loop), so a unit test in `FileService.crud.test.ts` is the correct and sufficient level; docker/chaos do not apply.

**Pre-existing latent bug found during review (NOT introduced by issue #8, not fixed here - out of scope)**: cross-restart orphaned-file recovery (`sshLite.reconnectOrphanedFile` + `detectOrphanedSshFiles`) is already dead code in the current temp-path layout. It depends on `parseHostInfoFromPath()` (`extensionHelpers.ts`) finding a literal `[SSH]` segment and reading the segment *before* it as the host hash. But (a) `getConnectionPrefix()` returns `'SSH'` only when the connectionId has fewer than 3 colon-parts, and real ids are always `host:port:username`, so files are `[user@host]`, never `[SSH]` - `parseHostInfoFromPath` returns null and recovery bails with "Cannot determine server from file path"; and (b) post-issue-#6 the segment before the filename is `<dirLabel>_<dirHash>`, not `<connHash>`, so even the hash it would extract is wrong. `none` mode (issue #8) does not regress this because the default `userAndHost` mode is equally affected. Proper fix (future): drive recovery off the `.sshLite-metadata.json` sidecar (`getRemotePathFromMetadata` already exists and is filename-independent) instead of parsing the filename.

## 2026-06-06 - Temp-file path must encode the remote FOLDER, not just the basename (issue #6); plus tooling gotchas

**Context**: GitHub issue #6 — two remote files with the same basename in different folders (`/var/www/domainA/index.php` and `/var/www/domainB/index.php`) opened the wrong content, and saving one could upload to the other. Root cause + reusable gotchas:

- **`getLocalFilePath` derived the temp path from `connHash + basename` only**, so same-named files on one server mapped to ONE temp file (`<tempDir>/<connHash>/[user@host] index.php`). The in-memory `fileMappings` (keyed by localPath), the on-disk `.sshLite-metadata.json` (keyed by basename), and VS Code's "already open" check all then collapsed both files together → wrong content shown, and `handleFileSave` could upload edits to the wrong `remotePath`. Fix: a per-remote-folder subdirectory `<tempDir>/<connHash>/<dirLabel>_<dirHash>/[prefix] <basename>` (hash of `path.posix.dirname(remotePath)`), and move metadata INTO that subdir so its basename key is unique per folder. ## 2026-06-06 - VS Code reveal() race: event-driven synchronization + IPC ordering (issue #7)

**Context**: GitHub issue #7 — "reveal in file tree selects the folder, not the file". Root cause: `revealFile()` called `refreshConnection()` (fires `onDidChangeTreeData`) then immediately returned; the caller called `fileTreeView.reveal()` before VS Code's renderer processed the tree refresh, so reveal selected the deepest item in the OLD tree (the folder).

**Fix approach**: event-driven synchronization with IPC ordering guarantee.

1. **`waitUntilVisible(connectionId, remotePath): Promise<void>`** — registers a resolver in `revealReadyCallbacks` keyed by `"connId:path"`. Returns a promise that resolves when `resolveRevealReady()` fires or after a 5-second safety timeout. Mark the timeout with `.unref()` (or `typeof timer === 'object' && timer !== null && 'unref' in timer` guard) so it doesn't prevent the Node process from exiting in tests.

2. **`resolveRevealReady(connectionId, remotePath)`** — fires the resolver for a specific path via `Promise.resolve().then(...)` (deferred microtask). This is the KEY mechanism: called in `revealFile()` AFTER `refreshConnection()` but BEFORE `await visiblePromise`. Because `refreshConnection()` sends its IPC to the renderer synchronously (or as a micro-batch), and `reveal()` is called only after the microtask resolves, the renderer always receives the refresh IPC BEFORE the reveal IPC — guaranteed by FIFO IPC ordering.

3. **`notifyVisible(connectionId, files: IRemoteFile[])`** — called from `getChildren()` as a secondary mechanism. Iterates the files returned by `getChildren()` and calls `resolveRevealReady(connectionId, file.path)` for each. Handles the case where VS Code calls `getChildren()` and the target file appears in the result BEFORE `resolveRevealReady()` fires from `revealFile()` (e.g., if VS Code is very fast).

4. **In `revealFile()` Case A and B**: call `waitUntilVisible()` before any loading, do all loading + `refreshConnection()`, then call `resolveRevealReady()` to release the `await visiblePromise`, then return `createFileTreeItem()`.

**Why not `setTimeout(N)`**: a fixed delay breaks on high-latency servers where N is too short.

**Why `resolveRevealReady()` from `revealFile()` not just `notifyVisible()` from `getChildren()`**: tests don't run VS Code so `getChildren()` is never triggered. Also, when the parent directory is EMPTY (`[]`), `notifyVisible([])` iterates nothing and never fires — the promise would hang until the 5s timeout.

**Test fix**: the `waitUntilVisible` callbacks are now resolved directly by `resolveRevealReady()` called from within `revealFile()` after all loading completes. Tests don't need fake timers or `getChildren()` simulation.

## 2026-06-01 — Upload showed the remote filesystem because the dialog had no local `defaultUri` — NOT because of host placement (corrected diagnosis)

**Applies to all OSes** (Windows/macOS/Linux clients equally). A user on **Remote-SSH** reported: SSH Lite **Download works** (file lands on their local machine) but **Upload** opens a picker showing the **remote server's** paths, not their own machine.

**My first diagnosis was WRONG and I shipped it.** I assumed the symptom meant SSH Lite was running on the **workspace (remote) extension host**, and built a whole "Install in Local" guidance story around it. The user then gave one decisive fact: *download works*. That single fact **disproves** the host-placement theory — if the extension were on the remote host, `showSaveDialog` would also land on the remote and download would NOT reach the local machine. Working download ⇒ the extension is on the **UI (local) host**. So the cause had to be something that makes *only the upload dialog* point at the remote while *download* points local, **with the extension on the same (local) host**.

**Actual root cause — asymmetric `defaultUri`.** A `showOpenDialog` / `showSaveDialog` with **no `defaultUri`** opens at the *window's* current folder. In a Remote-SSH window that folder is on the **remote server**, even when the calling extension runs on the UI (local) host. The dialog location follows the supplied `defaultUri`, not the extension host.

- `downloadFileTo` set `defaultUri: vscode.Uri.file(path.join(os.homedir(), name))` → a local `file:` path → dialog opens **local** → worked.
- `uploadFileTo` (v0.8.17) set `defaultUri: this.lastUploadUri`, which is **`undefined`** on first use → dialog fell back to the **remote workspace folder** → showed server paths.

**The fix is one line** (v0.8.18): seed a local default — `defaultUri: this.lastUploadUri ?? vscode.Uri.file(os.homedir())`. Now upload opens at the local home, exactly like download.

**Lessons** (carry forward):

- **A `file:` `defaultUri` is what pins a dialog to the local machine in a Remote-SSH window — not the extension host.** Any `showOpenDialog`/`showSaveDialog` that must target the user's own machine MUST pass an explicit local `file:` `defaultUri`. An `undefined` default silently follows the remote workspace. Audit every dialog call for a seeded local default; never rely on the implicit one.
- **One decisive contradicting fact beats a whole plausible theory — go back to root immediately.** "Download works but upload doesn't, same install" was incompatible with my host-placement story. The moment a user fact contradicts the diagnosis, discard the diagnosis AND everything built on it (I had written hint copy, a `warnRemoteWorkspaceUpload` gate, and docs around the wrong cause). The differential test for this class: **does Download land locally?** Yes ⇒ extension is local ⇒ the bug is in the *upload-specific* code (here: `defaultUri`), not host placement.
- **Host placement is a genuinely *different* scenario, kept separate.** If SSH Lite really IS installed on the workspace/remote host, then *both* download and upload browse the server and the remedy is "Install in Local" — that is real, but it was NOT this report. The `onRemoteWorkspaceHost` warning + broadened activation hint added in v0.8.18 only fire when `extensionKind === Workspace`, so they are harmless no-ops for the (common) local install and do not affect this fix.
- **Scheme-safe upload read was hardening, not the cause.** `readUserSelectedUri()` → `vscode.workspace.fs.stat`/`readFile` (replacing raw `fs.statSync`/`readFileSync(uri.fsPath)`) is correct defense-in-depth for non-`file:` URIs, but a local `file:` pick read fine under the old code — it did not cause this user's symptom. Don't let a real-but-secondary fix masquerade as the root cause.
- **Decode URI leaf names with a guard, never bare `decodeURIComponent`.** A parsed `vscode-remote:` path may be percent-encoded (`my%20file.txt`); a `file:` path may contain a *literal* `%` that is invalid encoding (`100%.txt`) and makes `decodeURIComponent` throw. `decodeUriComponentSafe` (in `utils/helpers.ts`) try/catches and falls back to the raw segment.
- **Reads before the `try/catch` must surface their own errors.** The upload read runs before `startActivity`; a failed `vscode.workspace.fs.readFile` would otherwise only be `logResult`'d by the caller with no user-visible message. Wrapped in its own try/catch → `showErrorMessage` + early return, matching the download path's UX.
- **Regression net**: `src/services/FileService.uploadUri.test.ts` asserts upload passes a **local `file:` `defaultUri`** when `lastUploadUri` is unset (the actual fix), plus the scheme-safe read across `file:` / `vscode-remote:` / `mem:` (asserts `vscode.workspace.fs.readFile` used, raw `fs.readFileSync` NOT), read-failure → `showErrorMessage`, and percent-decoded / literal-`%` leaf names.
- **Still-open identical scheme-safety gaps** (raw `fs` on a dialog `.fsPath`): `AuditService.exportLogs`, `keyCommands.pushPubkey`, `diffCommand`. Fix with the same `readUserSelectedUri`/`writeUserSelectedUri` + `decodeUriComponentSafe` shape — and seed a local `defaultUri` — when next touching those areas.

## 2026-05-22 — `fs.writeFileSync(uri.fsPath, …)` is unsafe; declare `extensionKind` when an extension belongs on the user's machine

**Applies to all OSes the user can run VS Code from** — Windows, macOS, Linux. The bug below is about *which extension host* the extension lands in, not the user's OS. The reporter happened to be on Windows; the same failure mode reproduces from a macOS or Linux client whose Remote-SSH target is a different machine.

**What happened**: A user on a client OS (Windows in the original report; same applies to macOS / Linux clients) used VS Code's built-in **Remote-SSH** extension to connect to a remote server, then installed SSH Lite from the Marketplace inside that session. Because `package.json` did not declare `extensionKind`, VS Code placed SSH Lite on the **workspace** extension host (running on the remote machine), not on the user's own machine. When the user clicked **Download** on a remote file:

- The save dialog defaulted to `/tmp/<vscode-tmp-id>/<name>` (a VS Code session-temp on the *remote* Unix-like host), not the user's actual home on their own machine.
- The download silently failed — `fs.writeFileSync(saveUri.fsPath, content)` either landed in a hidden VS Code temp dir or threw `EACCES`, depending on the Remote-SSH setup.
- The user expected behaviour like the PDF Viewer extension: an **Install in Local** button, with SSH Lite running on their own machine and the file landing in their own home (`C:\Users\<user>\...` on Windows, `/Users/<user>/...` on macOS, `/home/<user>/...` on Linux).

**Root cause** (two stacked layers):

- **Manifest layer**: `extensionKind` was undeclared. Marketplace falls back to "wherever VS Code prefers", which for a Remote-SSH session is the workspace host. SSH Lite was already designed as a UI-only client (all I/O via `vscode.SecretStorage`, `showSaveDialog`, `ssh2` native client; no `process.platform` branches; no local-file assumptions) but it never *told* VS Code that.
- **Code layer**: `FileService.downloadFileTo` and `downloadFolder` used `fs.writeFileSync(saveUri.fsPath, content)` / `fs.mkdirSync(folderUri.fsPath, ...)`. The dialog can return a URI whose scheme is **not** `file:` — on Remote-SSH workspace host it returns `vscode-remote://ssh-remote+host/...`, and `.fsPath` on that URI resolves to a path the raw Node `fs` module cannot reach (or writes to the wrong host's filesystem). The unit + integration suites only ever exercised `file:` URIs, so this latent bug shipped.

**Lesson** (mandatory checks for any future feature touching save/open dialogs OR extension hosts):

- **Declare `extensionKind` explicitly** for any extension whose semantic is "talk to remote servers from where the user actually sits". For SSH-client-type extensions like SSH Lite, `["ui", "workspace"]` (UI preferred, workspace allowed for chained SSH) is the right shape. Never rely on VS Code's default placement — it varies by VS Code version and remote backend.
- **Never call `fs.writeFileSync(uri.fsPath, …)`** (or `fs.readFileSync(uri.fsPath)`, `fs.mkdirSync`, etc.) on a URI returned by `showSaveDialog` / `showOpenDialog`. The URI scheme is **not** guaranteed to be `file:` — it can be `vscode-remote:`, `vscode-vfs:` (GitHub Codespaces, virtual workspaces), `untitled:`, or any registered `FileSystemProvider` scheme. Use `vscode.workspace.fs.{readFile,writeFile,createDirectory,delete,stat,readDirectory}` — they dispatch via the URI provider system and respect the scheme. `vscode.workspace.fs.createDirectory` is idempotent; drop the `if (!fs.existsSync(...)) fs.mkdirSync(...)` guard.
- **Use `vscode.Uri.joinPath(baseUri, ...segments)`** to build child URIs inside a folder dialog result. `path.join(folderUri.fsPath, name)` produces a *string*, dropping the scheme — when you later feed that string back into a URI you lose `vscode-remote:` / `vscode-vfs:`.
- **Surface a one-time hint** when an extension lands on the wrong host. Detection combo: `vscode.env.remoteName === 'ssh-remote'` plus `context.extension.extensionKind === vscode.ExtensionKind.Workspace`. Offer "Install in Local" + a dismissible "Don't show again" backed by a settings key (`sshLite.suppressLocalInstallHint`). "Download failed silently to a path I've never heard of" is a hostile experience; the user does not know which host the extension ran on.
- **Regression net**: every dialog-write path needs a unit test that exercises **at least three URI schemes** — `file:`, `vscode-remote:`, and one arbitrary custom scheme (`mem:`, etc.). The test asserts `vscode.workspace.fs.writeFile` was called AND `fs.writeFileSync` was NOT called on the dialog URI. See `src/services/FileService.downloadUri.test.ts` for the template.
- **Remaining dialog call sites with the same latent bug** (still using `.fsPath` with raw `fs`): `AuditService.exportLogs`, `keyCommands.pushPubkey`, `diffCommand`. (`FileService.uploadFileTo` read side was fixed in v0.8.18 — see the 2026-06-01 entry below.) Lower user impact but track as a known gap — fix when the next change touches that area.

## 2026-05-19 — `activate()` init steps MUST be wrapped in `safeStep`; an unguarded throw silently kills all tree views

**What happened**: A user on v0.8.10 reported all 4 SSH Lite tree views ("SSH HOSTS", "REMOTE FILES", "ACTIVITY", "PORT FORWARDS") showed *"There is no data provider registered"* and their saved hosts looked gone. We traced it to `src/extension.ts` `activate()`: it ran ~18 sequential init steps (service inits, virtual-doc providers, tree-provider constructors, then four `vscode.window.createTreeView` calls) **with no try/catch and minimal logging**. A single throw in any step — `credentialService.initialize`, `connectionManager.initialize`, `portForwardService.initialize`, `folderHistoryService.initialize`, `SnippetService.initialize`, or any tree-provider constructor — aborted the whole function before reaching `createTreeView`. Result: no view ever registered, no log to tell us which step failed, no way to recover without an extension reinstall. (Hosts were not actually deleted — they remained in VS Code User `settings.json` under `sshLite.hosts` — but with no tree to render them, the user thought they were lost and was on the verge of re-adding everything from memory.)

**Root cause**: VS Code's extension activation has no implicit safety net. If `activate()` throws, VS Code marks the extension as broken AND any `contributes.views` declared in `package.json` show the "no data provider registered" placeholder — which looks identical to "the tree provider isn't implemented yet" rather than "activation crashed". The user has no way to distinguish "my hosts were deleted" from "the extension never loaded". And without per-step logging, even reproducing the bug locally doesn't tell you which step is the culprit.

**Lesson — for any activation code added from v0.8.11 forward**:

- **Wrap each init step in `safeStep(name, fn)`** (defined at the top of `src/extension.ts`). The helper logs `lifecycle / activate/<name>-ok` on success, `activate/<name>-failed` with the error name + message + 3-line stack on throw, pushes the name onto `_activateFailures`, and returns `undefined` so subsequent steps still run. This is non-negotiable for service inits and `createTreeView` calls.
- **Tree-view variables become `TreeView<T> | undefined`** — guard every downstream usage (`if (view) { view.reveal(...) }`, `if (view) treeViewDisposables.push(view)` etc.). TypeScript will surface every missing guard via `'x' is possibly undefined` compile errors — fix them iteratively, don't skip with `!`.
- **Tree-provider constructors are intentionally NOT wrapped** because their downstream usages number in the hundreds (every command handler does `fileTreeProvider.refreshFolder(...)` etc.) and making the providers nullable would require guards in every handler. The bug in practice was in service inits, not provider constructors; if a provider ever does throw in the future, the throw propagates and VS Code marks the extension as broken — better than silently leaving the user with no UI.
- **End every activate with a summary**: if `_activateFailures.length > 0`, fire one `vscode.window.showErrorMessage` listing the failed step names so the user sees which feature is degraded and can open Output → SSH Lite for the per-step log. Always emit `infoLog('lifecycle', 'activate/complete', { failedSteps, failedNames })`.
- **Regression net**: `src/extension.activate.test.ts` asserts the happy path (4 trees register) AND the degraded path (one service throws, OTHER 3 trees still register, the failure is recorded, user gets one error notification). If a future refactor moves a throw-prone call outside `safeStep`, these tests fail.
- **The recovery path for users hit by an old version of this bug**: their saved hosts are still in `~/.../User/settings.json` under `sshLite.hosts`. The 0.8.11 release notes tell them where to look so they don't re-add everything by hand.

**Don't conflate "no UI rendered" with "data lost".** Always ask "did the storage layer actually delete anything, or did the rendering layer just fail?" before assuming the worst. SSH Lite saved hosts live in VS Code workspace configuration (`vscode.workspace.getConfiguration('sshLite').hosts` in `hostService.ts:154`), not in extension-managed JSON files — they survive even a totally broken extension.

## 2026-05-19 — Never transcribe a crypto address from a screenshot by eye; always decode the source QR

**What happened**: While building the donate section in `README.md`, I read the TON address from the wallet-app screenshot (`IMG_5389.png`) character by character — three times, even — and recorded it as `UQBbbIS1-F3ufPBPD13EKfp28G_A_j10kXNn-XuuxQUwoIEs` (uppercase `I` at position 6). v0.8.9 was committed and published with that string and a QR generated from it. User then asked me to verify carefully against the screenshots again, this time by decoding the source QR with `jsqr`. The QR actually encoded `UQBbblS1-…` — lowercase `l`, not uppercase `I`. The published donate page would have routed TON donations to a *different valid TON address* (or nowhere) — irrecoverable funds.

**Root cause**: iOS's sans-serif (SF Pro) renders uppercase `I`, lowercase `l`, and digit `1` as essentially the same vertical stroke at small sizes. Same applies to `0`/`O`/`o`, and `B`/`8` in some fonts. Visual transcription cannot reliably distinguish them. I performed the "triple-check" but all three checks used the same flawed input (my eyes on the same pixels), so they all returned the same wrong answer with growing confidence.

Compounding factor: my generator script then *encoded* my mistranscribed string into a QR, and when I "verified" by decoding that generated QR with `jsqr`, of course it round-tripped correctly — I was checking that the QR matched my string, not that my string matched the source. That's a tautology, not a verification.

**Lesson**:

- **For any address that came from a QR code (crypto wallets, payment links, anything money- or auth-critical):** decode the source QR with `jsqr` (or another decoder) and use the decoded bytes as ground truth. Never transcribe by eye, no matter how many times.
- **"Verification by re-encoding then decoding" is a tautology.** It only proves the generator round-trips, not that the encoded value matches the source. The verification chain must be: `source screenshot → decode → string A`; `our string → string B`; assert `A === B`.
- **Watch for these character pairs in any base64/base64url payload** (the alphabet TON, GitHub gist IDs, JWTs, etc. use): `I`/`l`/`1`/`i`, `O`/`0`/`o`, `B`/`8`, `S`/`5`, `Z`/`2`. base58 (used by Solana, Bitcoin) deliberately excludes most of these (`0`, `O`, `I`, `l`) — that's a feature, not an accident. Hex (used by EVM chains) is also unambiguous.
- **When a verification step succeeds, ask "what input did this verification consume?"** If the answer is "the same input I'm trying to verify", it's not a verification.

## 2026-05-07 — Tab state defaults must be overwritten by every payload that carries the value

**What happened**: User configured `sshLite.searchMaxResults: 10000`. The search correctly returned 10000 results (extension-side cap was honored), but the UI banner read "⚠️ Limit 2000 reached" instead of 10000. The displayed value did not match the configured setting.

**Root cause**: `createTabState()` initializes `tab.limit: 2000` as a hardcoded default. The `'searchBatch'` message handler (and its kept-tab branch) updated `results`, `hitLimit`, and `searching` on each batch but never assigned `tab.limit = message.limit`, even though every batch payload carries the extension's `maxResults` value. So `tab.limit` stayed at 2000 forever, and `renderResults()` read `tab.limit || limit` and rendered "2000" in the warning text. Behavior was correct (10000 results stopped the search at the right point); only the displayed cap value was stale. The `'results'` handler (a different message path) did update `tab.limit`, which is why the bug was inconsistent and easy to miss.

**Lesson**:
- When a tab/state object has fields with hardcoded defaults (`limit: 2000`, `viewMode: 'list'`, etc.), audit every message handler that produces or refreshes that state. Each handler must overwrite every default-bearing field whose canonical value is in the payload — not just `results`/`hitLimit`/`searching`. A field that's only set in *one* of several handlers will display the default everywhere else.
- For settings that surface in user-visible text (limit warnings, banner counts), the source of truth should be the live payload, not a tab-state default. Default fields should be invariant skeletons, not values used as fallbacks.
- When a config value flows extension→webview, run a focused check: search for everywhere `tab.<field>` is read in the renderer, and confirm every message handler that affects that tab updates `<field>` from the payload.

## 2026-05-07 — `vsce package --no-dependencies` strips ssh2 and breaks activation

**What happened**: The Phase 1 `scripts/verify-package.js` ran `vsce package --no-dependencies`. The flag tells vsce to omit `node_modules/` from the .vsix. The 4-entry `REQUIRED_ENTRIES` check verified `media/search/*` and `out/extension.js` were present and reported success. But the shipped .vsix was missing the entire `node_modules/` tree — including `ssh2` and `ssh-config`, both runtime dependencies. On install, `require('ssh2')` failed at module load, `activate()` never ran, and every tree view showed "no data provider registered for this view." The user had to surface the error from the DevTools console; the verify-package smoke test gave a green light.

**Root cause**:
- `--no-dependencies` is appropriate ONLY when an extension uses a bundler (esbuild/webpack) to inline all runtime deps into a single output file. SSH Lite does not bundle the extension itself — only the search webview is bundled. The extension's `out/extension.js` still does `require('ssh2')` at runtime.
- The verify script's required-entries list checked the webview bundle but not runtime deps. A green check on the webview said nothing about activation viability.
- The shipped .vsix was 1 MB instead of the expected ~5 MB. A size sanity check would have caught this immediately.

**Lesson**:
- `vsce package` (no flags) is the correct release command for SSH Lite. The standard behavior bundles `dependencies` from `package.json` (not `devDependencies`) and that is exactly what we want.
- Any packaging-smoke script must verify production deps ship: include `extension/node_modules/<runtime-dep>/package.json` in REQUIRED_ENTRIES for every entry in `package.json`'s `dependencies`. Currently: `ssh2`, `ssh-config`.
- A .vsix sanity-size check is cheap and effective: SSH Lite's expected size is ≥ 4 MB. Anything substantially smaller indicates missing deps.
- Reviewers cannot catch packaging bugs by reading code — they manifest only at install/runtime. The packaging-smoke script IS the test; its assertions must be specific enough to fail on broken artifacts.

## 2026-05-07 — Lifting JS out of a template literal needs a THREE-step unescape, not two

**What happened**: Phase 1 of the search-render-overhaul (v0.8.1) lifted the inline `<script>` body from `SearchPanel.getWebviewContent()` (a template literal) into `webview-src/search/index.ts`. The unescape pass handled `` \` → ` `` and `\${ → ${` but missed `\\u → \u` (and `\\u{` → `\u{`). After the lift, every emoji and special character (server status icons 🔄 ❌ 🟢 ⚡ ⚫, path icons 📄 📁, warning ⚠️, remove/close ×, sort ↑, tooltip em-dashes —) rendered as literal escape text like `\u{1F504}` in the UI. Three reviewers (Task 6 spec + quality, then a holistic Phase 1 review) examined the lift; only the holistic final review caught this. Manual smoke test (Task 11) was skipped per user request — that gate would have caught it in 30 seconds.

**Root cause**: A string literal inside a template literal is double-evaluated: once by the template literal itself, then again by the inner JS engine when the template's string is loaded as code. So the original `'\\u{1F504}'` in the template-literal source produced `'\u{1F504}'` at template-output time, which the webview's JS engine then parsed as the emoji. After lifting to a real `.ts` file, only one level of evaluation happens — `'\\u{1F504}'` becomes literal text `\u{1F504}` (9 chars), never interpreted as a unicode escape.

**Lesson**:
- When lifting any JS body out of a template literal into a real source file, the unescape pass must include **all** template-literal escape sequences, not just backticks and `${`. At minimum: `` \` ``, `\${`, `\\u`, `\\u{`, `\\xHH`, `\\n`, `\\r`, `\\t`, `\\\\` (literal backslash). Run a grep for `\\\\` in the source after the lift; every hit is a candidate for the same regression.
- Per-task code reviewers can miss UI-rendering bugs because they don't run the UI. The **manual smoke test** is the only reliable gate for this class of bug. If the user opts to skip manual smoke, flag the unicode/escape-sequence risk explicitly in the report so they can run a targeted check (e.g. open the panel and verify icons render).
- Add a one-liner check to any future webview-lift plan: `grep -nE '\\\\[a-zA-Z0-9{]' webview-src/<dir>/index.ts` should return zero hits after the unescape; any hit is a regression risk.

## 2026-06-06 — Keyboard-invoked tree commands DON'T receive the selection as an arg (the "shortcut does nothing" bug)

**What happened**: After adding keyboard-shortcut hints to tooltips, the user reported the file-row shortcuts (Ctrl+C / Ctrl+X / F2 / Ctrl+V → copyRemoteItem / cutRemoteItem / renameRemote / pasteRemoteItem) "don't work — I press the key and nothing happens." Root cause: those handlers were written for context-menu invocation, where VS Code passes the right-clicked tree item as the first argument. When the SAME command is invoked via a **keyboard shortcut**, VS Code passes **no arguments** — it does NOT auto-pass the tree view's selection. So every handler hit its `if (!item) return;` guard and silently no-op'd. The keybindings existed in package.json and the `when`/context keys were correct; the handlers just had no target.

**Lesson**:
- A `view/item/context` command handler that only reads its `item` argument is BROKEN when the same command is given a keybinding. Keyboard invocation passes nothing.
- Fix pattern: fall back to the tree view's live selection. Capture the `vscode.TreeView` from `createTreeView` and, when no arg is passed, read `treeView?.selection`:
  ```ts
  const fromArgs = (items?.length ? items : (item ? [item] : []));
  const source = fromArgs.length ? fromArgs : (fileTreeView?.selection ?? []);
  const fileItems = source.filter((i): i is FileTreeItem => i instanceof FileTreeItem);
  ```
  For single-target commands (rename/paste): `const target = item ?? (fileTreeView?.selection ?? []).find(i => i instanceof FileTreeItem)`.
- The view must have `canSelectMany`/focus for `.selection` to be populated; the keybinding `when` (`focusedView == sshLite.fileExplorer && listFocus`) already guarantees the tree is focused when the key fires.
- **Focus-steal caveat**: `FileTreeItem` sets `command: openFile`, so single-clicking a FILE opens it and moves focus to the editor — then `focusedView == sshLite.fileExplorer` is false and Ctrl+C/F2 act on the editor, not the tree. The reliable gesture is to keep focus in the tree (arrow-key navigation does NOT trigger the open-on-click), or use a folder row. Worth knowing before claiming "the shortcut works" — it works only while the file explorer has keyboard focus.
- **Verification gotcha (the real miss)**: I originally shipped the tooltips and claimed done after testing only that the tooltip STRING rendered — never that the shortcut FIRES. You cannot prove a keybinding works from a string-render test. Prove it by invoking the REAL registered handler with no arg through `executeCommand(id)` (the vscode mock's `executeCommand` dispatches to the real registry) with a controlled `treeView.selection`, and assert the operation happened. See `src/extension.keyboardShortcuts.test.ts`.
- The 6 view-level shortcuts (connect, openTerminal, refreshFiles, goToPath, filterFiles, showSearch) were already fine: they either need no target or fall back to a quick-pick when invoked with no arg.

## 2026-06-06 — SSH Lite shortcut scheme: global two-key chord with dual leaders + first-letter keys

**Final design** (after iterating through several rejected forms — record the destination AND the dead-ends):
- The **view-level** commands use a **two-key chord** with **two interchangeable leaders**: `Ctrl+Alt+D` (left-hand, home row) and `Ctrl+Alt+K` (right-hand) — mac `Cmd+Alt+D` / `Cmd+Alt+K`. The second key is the command's **first letter**. So Connect = `Ctrl+Alt+D C` *or* `Ctrl+Alt+K C`, etc. Each command has 2 keybinding entries (one per leader). The covered set grew over the session to 15 global view-level commands (C connect, T terminal, R refresh, G go-to-path, F filter, S search, A add-host, H home, U up/parent, P port, M monitor, E env, B batch, Q quick-status, K gen-key) → 30 chord entries + 5 file-row (F2/Ctrl+C/Ctrl+X/Ctrl+V/Delete) = 35 total.
- **Leader must not be a global OS hotkey**: `Ctrl+Alt+S` was rejected as a leader because a common screenshot tool registers it as a system-wide hotkey, and the OS intercepts global hotkeys *before* VS Code sees them — so VS Code can never override it and the chord would never start. Any `Ctrl+Alt+<letter>` can be grabbed by an installed app; the user is the authority on which letter is free on their machine. `Ctrl+Alt+D` was the chosen free left leader.
- These are **global** (`when: sshLite.hasConnections`, except `connect` which has no `when`). They are NOT scoped to panel focus — that was the key correction.
- File-row ops (F2 / Ctrl+C / Ctrl+X / Ctrl+V) stay tree-scoped (`focusedView == sshLite.fileExplorer && listFocus`) because they act on a selected remote row; there is no such target while the editor is focused.

**The journey (each step was rejected for a concrete reason — don't repeat them)**:
1. `Ctrl+Shift+<letter>` (original): collided with VS Code defaults (Find in Files, Save As, Reopen Closed Editor, Source Control).
2. Single `Ctrl+Alt+<letter>`, scoped to view focus: **wrong because not global** — the user is usually in the editor when they want to refresh/search/connect, and a focus-scoped binding does nothing there. *The scoping that makes a binding "safe" also makes it useless for global actions.*
3. `Alt+<letter>` (single): **bad on Windows** — bare `Alt+<letter>` triggers menu-bar mnemonics (Alt+S=Selection, Alt+T=Terminal, Alt+R=Run, Alt+G=Go, Alt+F=File...).
4. Two-key chord (final): a chord prefix never shadows any single common shortcut, so it can be global AND conflict-free. Two leaders give both-hand ergonomics; the right-side `Ctrl+Alt+K` leader keeps the left-side first-letter keys comfortable (hands alternate).

**Conflict reality to remember**: no combo is "reserved" on the Marketplace. VS Code resolves by `when` + load order; a view-scoped binding wins in-context and doesn't break other extensions' global bindings. So cross-extension conflict is rarely the real constraint — *focus scope* (does it fire where the user is?) and *menu mnemonics / AltGr* are.

**Mechanics**:
- Only `package.json` `contributes.keybindings` changed. Tooltips update automatically because `src/utils/keybindingHints.ts` reads the shortcut live from the manifest and `prettifyKey` renders chords (`ctrl+alt+d c` → `Ctrl+Alt+D C`); `getShortcut` returns the first binding, so tooltips show the `Ctrl+Alt+D` leader. No tooltip code touched.
- `docs/COMMANDS.md` is generated; its hardcoded shortcut mentions live in `scripts/generate-commands-doc.js` (not the .md). Update the generator, then `npm run docs:commands`.
- swc/jest does NOT type-check, so a test that violates a required interface field (e.g. `IRemoteFile.connectionId`) passes under jest but fails `tsc`/`npm run compile`. Always run `npx tsc -p ./ --noEmit` after adding test fixtures, not just jest.

## Reveal-in-tree (issue #7): getParent() must resolve `~` before comparing paths

**Symptom**: "Reveal in File Tree" navigates to the folder but does NOT select the file. Hits the DEFAULT home (`~`) view.

**Two independent root causes** (fix both):
1. **getParent() `~` mismatch (main cause).** SFTP resolves `~` via `realpath('.')`, so `listFiles('~')` returns ABSOLUTE child paths (`/home/user/...`). But `currentPath` stays the literal `'~'`. `getParent()` compared an absolute `dirname(file.path)` against `'~'`, never matched the root, and emitted phantom `/home`, `/home/user` nodes that are NOT rendered in the tree. VS Code's `TreeView.reveal()` walks `getParent()` to build the ancestor chain and matches each node by `TreeItem.id`; a phantom node it can't find as a child aborts the walk, so the file is never selected. **Fix**: `resolveCurrentPathSync()` expands a leading `~` using a cached absolute home (from `echo ~` in revealFile, or derived from the cached `~` listing's `dirname`), and getParent compares `parentPath === rootAbs`. See `FileTreeProvider.ts` getParent + `resolvedHomePaths`.
2. **Timing race.** Old code revealed after a fixed `setTimeout(300)`; on a slow link the folder was not listed yet so reveal() silently no-oped. Fix: `waitUntilVisible()` makes revealFile() resolve only after the children are in the tree model.

**Test levels**: unit `FileTreeProvider.test.ts` "issue #7" (getParent chain has no phantom nodes at `~`); integration `docker-ssh-reveal.test.ts` (real laggy server, real SFTP-resolved home).

**Gotcha for reveal()**: any TreeDataProvider with a synthetic `getParent()` (constructing parent items rather than returning cached refs) MUST produce ids that match what `getChildren()` actually renders at the current root. "Same relative slot" is not enough; the chain depth must match the rendered depth.

## FTP listFiles must confirm existence on an empty result (SSH parity) + pure-ftpd cold-start warmup (issue #9)

Two findings while stress-testing FTP against live containers (vsftpd 2207 + pure-ftpd 2208):

1. **basic-ftp swallows 550 on LIST of a missing dir.** `client.list('/missing')` resolves to `[]` on BOTH vsftpd and pure-ftpd instead of throwing, unlike SFTP `readdir` which throws "No such file". So a deleted/renamed folder rendered as a misleading empty folder over FTP while erroring over SSH. Fix: `FTPConnection.listFiles` confirms existence ONLY when the listing is empty (via `statRaw`, which lists the parent + matches basename, short-circuiting home/root with no network call) and throws `FTPError('No such directory')` only when it can PROVE absence. Conservative on failure: if the existence probe itself throws (unreadable parent on a non-chrooted server like vsftpd 2207), trust the successful empty listing and return `[]` rather than surface a misleading error. Cost: one extra round-trip only for empty/missing dirs; populated dirs and home/root cost nothing. Also normalize `_homePath` (strip trailing `/` from `pwd()`) so the home short-circuit compares cleanly on servers that append a slash.

2. **pure-ftpd blocks ALL connections (plain AND FTPS) for ~30-90s on first boot** while it generates a self-signed cert + 2048-bit DH params ("Generating DH parameters ... going to take a long time"). A cold container answers connect with "Server sent FIN packet unexpectedly" (plain) or a TLS handshake error (FTPS). vsftpd (2207) has no such delay. The FTP docker suites are a MANUAL prerequisite (`docker compose up -d ftp ftp-pure`) that globalSetup does NOT manage or warm, so running immediately after `up` flaked. Fix: shared `connectWithRetry` (`src/integration/ftpTestHelpers.ts`) retries connect for up to ~90s in every FTP suite `beforeAll` (with the jest `beforeAll` timeout bumped to 120-130s). Warm containers connect first try, so it costs nothing once running. Proven from a COLD start: all 71 docker FTP tests pass, pure-ftpd suites taking ~77s (waiting out warmup), vsftpd-only fileops in ~7s.

Process notes: (a) the "A worker process has failed to exit gracefully" warning is a GENERAL jest artifact in this repo (it also appears in the no-docker unit suite of 1832 tests), NOT an FTP socket leak: `--detectOpenHandles` (which forces runInBand) on the FTP suites reports ZERO open handles, and `FTPConnection.disconnect()` closes the control socket. (b) A wrong test assumption ("listFiles of a missing dir throws") surfaced the real behavior; per the plan-first rule, the wrong assumption was discarded and the actual contract was decided deliberately rather than patched. (c) `statRaw` runs on the already-held client INSIDE the same enqueued op, so the second `client.list` does NOT violate basic-ftp's one-command-at-a-time constraint.

## SSH-only features must be capability-gated in CODE, not just menus, or they crash on FTP (issue #9)

FTPConnection implements only the file-ops subset of IConnection; it has no `exec`, `shell`, `forwardPort`, `stopForward`, `client`, `readFileChunked`, `readFileLastLines`, search, sudo, native watch, or server backup. ConnectionManager hands connections out typed as `SSHConnection` via a downcast (`asSSH`), so the COMPILER does not catch an SSH-only call landing on FTP - it throws a runtime "x is not a function" TypeError. Menu `when` clauses gate FTP rows with `(?!\.ftp)`, but that ONLY hides menu items: keybindings, the Command Palette, connection pickers, the active-connection path, and auto-restore all bypass menu gating and reach the handler/service directly.

A four-agent audit found ~40 reachable crash sites: openTerminal/openTerminalHere (shell), monitor/quickStatus/diagnoseSlowness (exec), all SSH Tools (snippet/cron/env/process/service/key/batch via exec, reachable from the palette through `pickConnection`), searchInScope/showSearch/searchServerForFilter/indexFolder + per-folder filename filter (search), forwardPort/stopForward, FileService server-backup x9 + large-file preview/view + properties + progressive download/preview. `selectConnection` and the SSH Tools `getConnectedConnections` picker offered FTP connections to shell-only commands. The per-folder `filterFileNames` inline icon is even shown ON FTP rows (canonical inline slots use `(\.ftp)?`), so it was a menu-reachable crash.

Fix = a two-layer guard (`src/utils/capabilityGuard.ts`):
1. `ensureCapability(conn, cap, action?)` at command-handler entry - shows a friendly warning and returns false (early-return). Plus FILTER the pickers: `selectConnection(cm, requireCapability?)`, SSH Tools `getConnectedConnections` (require supportsExec), `PortForwardService.promptForwardPort`, and exclude FTP hosts from `buildServerSearchEntries` (cross-server search list).
2. `assertCapability(conn, cap, action?)` at the SERVICE SINK (throws a clear Error) as a backstop, so the SSH-only method is never invoked even via a path the handler guard missed: CommandGuard.openShell/exec/searchFiles, TerminalService.createTerminal, PortForwardService.forwardPort, SystemToolsService.*, SshKeyService.pushPublicKey, FilenameIndexService.buildIndex, FileService backup/large-file/properties methods, ProgressiveDownloadManager/ProgressiveFileContentProvider.

CRITICAL gotcha: `hasCapability` must be LENIENT - block ONLY when a capability is explicitly `false` (real FTP), and treat a connection with NO `capabilities` object as capable. Real SSHConnection/FTPConnection always populate `capabilities`; the only objects without it are legacy/test stubs, and the repo convention is "unknown connection kind defaults to SSH". A strict `!!caps?.[cap]` broke 66 existing tests whose mock connections have no `capabilities` (they got filtered out / threw). With the lenient rule, mocks pass and only real FTP is blocked.

Regression tests: `src/utils/capabilityGuard.test.ts` (helper) and `src/__tests__/ftp-capability-guards.test.ts` (feeds each guarded sink an FTP-capability object and asserts it throws `/not available over FTP/i` - proving the SSH-only method was never reached, i.e. no TypeError). The docker-ftp suites prove the SUPPORTED file ops still work. Note: search/exec calls that go DIRECT on the connection (FileTreeProvider, SearchPanel, FilenameIndexService, ServerMonitorService, FileService) bypass CommandGuard, so guarding CommandGuard alone is NOT enough - each direct sink needs its own guard.

## FTP correctness gotchas: empty-password discrimination + control-socket drop detection (issue #9)

Two `FTPConnection` defects found in a follow-up audit and fixed (each with a unit regression in `FTPConnection.test.ts`):

1. **`!password` discards a legitimately-empty stored password.** `_resolvePassword`'s stored-credential arm used `if (!password)` to decide "no secret -> prompt", but `getCredentialSecret` returns `''` for a saved EMPTY password (valid on some FTP accounts) and `undefined` only when truly absent. `''` is falsy, so the user was re-prompted every connect and a valid empty secret was thrown away. Fix: discriminate on `=== undefined` (mirror the no-stored-credential arm, which already did). RULE: when a stored value can be a legitimately-empty string, test for `undefined`/`null`, never truthiness. Same trap applies to the post-prompt check (`showInputBox` returns `''` for empty-typed vs `undefined` for Escape).

2. **A cached control-socket `close` listener misses a swapped socket.** `attachCloseHandler` caches `client.ftp.socket` once at connect and attaches `.once('close')`. basic-ftp's `socket` is a GETTER returning the CURRENT socket and it calls `removeAllListeners()` on the old socket when it swaps it (explicit-FTPS `AUTH TLS` upgrade, or a reconnect), so after a swap the listener is on a discarded object and a server-initiated drop never flips state to `Disconnected` - the connection sits "Connected" forever until the next op throws. Fix: ALSO reconcile state from the failed-operation path in `runOp`'s catch - basic-ftp exposes `client.closed` (true after any timeout/connection error), so `if (client.closed && state===Connected && !_closing) setState(Disconnected)`. RULE: don't rely solely on a one-time cached-socket event for drop detection on a library that can swap its socket; treat a failed op on a `closed` client as the authoritative drop signal. Verified: all 71 live docker FTP tests (vsftpd 2207 + pure-ftpd 2208 incl. FTPS) still pass.

3. **Companion gating fixes shipped same pass:** added `assertCapability(connection,'supportsExec'|'supportsShell')` to every `ServerMonitorService` entry method (the one exec-bearing service that lacked a sink backstop - safe today but inconsistent), and passed `'supportsServerBackup'` to `selectConnection` in the `showAllBackups`/`openServerBackupFolder` pickers so FTP is filtered out of the palette picker (it previously appeared then failed at the sink). Regression: `ftp-capability-guards.test.ts` now feeds an FTP connection to all nine ServerMonitorService methods and asserts each throws `/not available over FTP/i` (no TypeError).

## Auth UX: password prompt must be a LAST RESORT, not an unconditional fallback (connect button)

Reported bug: after setting a private key path on a host, the FIRST connect still popped a "Password for user@host" box. Root cause: there are TWO connect entry points and they reach `SSHConnection.buildAuthConfig` differently. `sshLite.connectWithCredential` (extension.ts:3341) builds a `privateKey` SavedCredential and hits the FIRST branch of buildAuthConfig (`if (this._credential)`), which for `privateKey` never asks for a password. But the plain `sshLite.connect` tree button (extension.ts:909) calls `connectionManager.connect(host)` -> `createConnection(host)` with NO credential -> `_credential === undefined` -> the LEGACY branch of buildAuthConfig (SSHConnection.ts ~564-622). That branch read the key correctly AND then UNCONDITIONALLY ran `creds.getOrPrompt(id,'password',...)` ("always include for fallback"), so a password box appeared even though key auth was already set up.

Fix (SSHConnection.ts): in the legacy branch, only PROMPT for a password when there is no other auth method — `haveKey || haveAgent ? await creds.get(id,'password') : await creds.getOrPrompt(...)`. `creds.get` returns a saved password without prompting, so a key/agent host gets a silent password fallback (true backward compat for users who had both) but is never bugged. Encrypted-key passphrase prompting (lines 575-578) is untouched — passphrase != login password.

LESSONS:
1. When a setting (privateKeyPath) is supposed to change auth behaviour, trace EVERY connect entry point — a fix/feature applied only to `connectWithCredential` silently misses the main `connect` button which goes through a different (legacy) code path. Grep all callers of `connectionManager.connect` vs `connectWithCredential`.
2. "Always include for fallback" by PROMPTING is an anti-pattern: a fallback you collect up-front by interrupting the user defeats the primary method's UX. Collect fallbacks silently (saved value), prompt only when nothing else can authenticate.
3. Test-mock gap caught the change: `SSHConnection.logs.test.ts` does NOT mock `fs` (reads the dev machine's real `~/.ssh/id_*`) and its CredentialService mock lacked `get`. On a dev box WITH a default key, `haveKey` became true and the new `creds.get` call hit an undefined mock method. When adding a new CredentialService method call in production, add it to every test mock of CredentialService (grep `jest.mock('../services/CredentialService'`). Regression unit test: `SSHConnection.auth.test.ts` (mocks fs + CredentialService) asserts: key present -> no password prompt; key+saved password -> attached silently, no prompt; agent only -> no prompt; no key/no agent -> prompt; encrypted key -> passphrase prompt only.

## Auth UX part 2: prompt for a key passphrase ONLY when the key is actually encrypted (detect via ssh2.utils.parseKey)

Follow-up to the password-prompt fix. After the login-password box was suppressed for key hosts, a SECOND prompt was exposed: the `sshLite.connectWithCredential` command (the connect icon on a user row) ran `vscode.window.showInputBox({ prompt: 'Passphrase for <path> (leave empty if the key has no passphrase)' })` UNCONDITIONALLY whenever the host had a `privateKeyPath` and no saved credential yet (extension.ts ~3373). So an UNENCRYPTED key (the common case, e.g. a passwordless `id_ed25519`) still popped a passphrase box on first connect.

Root issue is "prompt for something you don't need". Fix: only prompt when the key is genuinely encrypted, decided by `isPrivateKeyEncrypted()` in the new `src/connection/keyEncryption.ts`. It calls **ssh2's own `utils.parseKey(keyData)`** — the exact parser the live connection uses — which RETURNS (does not throw) an Error "Encrypted private OpenSSH key detected, but no passphrase given" for an encrypted key with no passphrase, and parses fine for an unencrypted one. We match `/passphrase|encrypted/i` on the error message; any other parse failure (malformed key) returns false so the real connection error surfaces instead of a misleading passphrase prompt.

This ALSO fixed a latent correctness bug in the legacy `buildAuthConfig` path: it used `privateKey.toString().includes('ENCRYPTED')`, which only matches legacy PEM headers (`Proc-Type: 4,ENCRYPTED`). A modern OpenSSH-format encrypted key (`ssh-keygen` default; cipher e.g. `aes256-ctr` lives INSIDE the base64 body, no "ENCRYPTED" text) was misdetected as unencrypted -> no passphrase collected -> ssh2 connect failed. Both legacy-path checks now use `isPrivateKeyEncrypted`. Verified empirically: `node -e` with ssh2 parseKey on a real encrypted ed25519 returns the encrypted Error; the key text contains no "ENCRYPTED".

LESSONS:
1. `ssh-keygen` produces NEW OpenSSH format by default — encryption is a cipher name in the binary body, NOT a `Proc-Type: ENCRYPTED` header. Never sniff key encryption with a substring match; use a real parser (ssh2 `utils.parseKey`) so your decision matches what the connection layer will actually do.
2. When suppressing one unnecessary prompt, look for EVERY other one in the same family — THREE separate sites each had their own unconditional passphrase prompt: the legacy `buildAuthConfig` path (`sshLite.connect`), the `connectWithCredential` first-connect credential setup, and `promptPrivateKeyAuth` (the "add user with key" command). Fixing one left the other two visible. Grep all `showInputBox` prompts containing "Passphrase"/"Password" and trace every connect entry point (`connect`, `connectWithCredential`, `connectToPinnedFolder`, auto-reconnect, add-user) before declaring an auth-UX fix complete. `connectToPinnedFolder` and auto-reconnect reuse an EXISTING credential (no prompt), so they were already safe.
3. Test mocks: a new shared module imported by `SSHConnection.ts` (`./keyEncryption`) breaks tests that `jest.mock('ssh2')` with only `Client` (no `utils`). Mock the new module directly in those suites (`jest.mock('./keyEncryption', () => ({ isPrivateKeyEncrypted: () => false }))`) instead of widening the ssh2 mock. Real-parser coverage lives in `keyEncryption.test.ts` (unmocked ssh2 + embedded throwaway fixture keys, incl. a modern-format encrypted key proving the old substring heuristic would have failed).

## Auth UX part 3: a wrong INITIAL passphrase must surface a clear notice (ssh2 reports it as a client-side parse error, not a server auth error)

A wrong/empty/missing key passphrase does NOT come back as a server auth rejection — ssh2 fails to decrypt the key locally and throws a client-side `Cannot parse privateKey: ...` error. The `connectWithCredential` catch handler classified auth failures only by server keywords (`isAuthError` = /authentication|password|Permission denied|All configured authentication methods failed|Invalid username/). None of those appear in the parse error, so a wrong passphrase fell through to the generic `else` branch: an opaque "Connection failed: Cannot parse privateKey: OpenSSH key integrity check failed -- bad passphrase?" with NO "Re-enter Passphrase" option.

The three exact ssh2 messages (captured live against the encrypted docker key on port 2216):
- wrong:   `Cannot parse privateKey: OpenSSH key integrity check failed -- bad passphrase?`
- empty:   `Cannot parse privateKey: Failed to generate information to decrypt key`
- missing: `Cannot parse privateKey: Encrypted private OpenSSH key detected, but no passphrase given`

Fix: `isKeyPassphraseError(message)` in `src/connection/keyEncryption.ts` matches `/bad passphrase|integrity check|decrypt key|no passphrase given/i` (passphrase-specific signals — a malformed non-encrypted key fails with a different `Cannot parse privateKey` detail that intentionally does NOT match, so it is not mislabeled). The catch handler now enters the privateKey recovery branch on `isAuthError || isPassphraseError`, and for the passphrase case shows a clear notice: `Incorrect passphrase for private key "<path>". The passphrase you entered could not decrypt the key.` plus a `Re-enter Passphrase` action; the retry's own failure is likewise specific (`Still an incorrect passphrase ...`).

LESSONS:
1. Client-side key-decrypt failures and server-side auth rejections are DIFFERENT error classes with disjoint message vocabularies. A keyword sniff tuned for server auth ("Permission denied", "authentication") silently misses a wrong passphrase. Classify both.
2. Don't show users the raw `Cannot parse privateKey: ...` string — translate the known passphrase failures into a plain-language notice with the actionable recovery (re-enter), and keep the raw text only for unclassified errors.
3. Capture the ACTUAL error strings from a real server (docker, encrypted key) before writing the matcher — guessing the wording is how these classifiers drift. Verified end-to-end: real ssh2 error -> compiled `isKeyPassphraseError` -> true for all three cases. Unit coverage in `keyEncryption.test.ts` pins the exact strings.

## Auth UX part 4: file picker (mouse) for the private-key PATH; recovery re-prompts password/passphrase keeping the key path

Two related UX asks. (a) Typing the private-key path by hand is clumsy — users want to browse and click the key file. (b) After a connect with saved/default credentials fails, the user must be able to re-enter the password or passphrase (not just see an error).

(a) New shared helper `src/utils/keyFilePicker.ts` → `pickPrivateKeyPath({ title, initialPath?, optional? })`. It shows a QuickPick with "Browse for key file…" (opens `vscode.window.showOpenDialog`, defaulting to `~/.ssh` when it exists) and "Type the path (supports ~)" (the old `showInputBox`, kept so power users retain `~`-portable paths). When `optional`, it also offers "No key — use password auth" and treats an empty answer as valid (returns ''). Return contract: non-empty string = chosen path; '' = explicit "no key"; undefined = cancelled. Wired into ALL three key-path entry points — HostService add-host (~666), HostService edit-host (~824), and `promptPrivateKeyAuth` (the "add user with key" command). Browse returns an absolute fsPath (loses `~`); Type preserves `~` — that is the intended trade-off (user picked the input mode). Cancelling on the optional host flows maps to "no key" to preserve the legacy non-aborting behaviour.

(b) Recovery already keeps the key path and re-prompts: the `connectWithCredential` catch handler offers "Enter New Password" (password creds) or "Re-enter Passphrase" (privateKey creds) on auth/passphrase failure, then `updateCredentialPassword` + reconnect. The key PATH is never re-asked there (user confirmed: keep key path, only re-enter the secret) — so no file picker in the recovery step, only in the key-path setup steps.

LESSONS:
1. For any file/path the user must supply, offer BOTH a browser (showOpenDialog, mouse) AND manual typing — a pure file dialog loses `~`-portable paths and quick keyboard entry; a pure text box is clumsy for mouse users. A QuickPick that branches to either is the cleanest VS Code pattern. Centralise it in one helper so add/edit/other flows stay identical.
2. Separate "where do I get the credential" (path/secret SETUP) from "the secret was wrong" (RECOVERY). Recovery should re-ask only the thing that can be wrong (the secret), keeping stable inputs (the key path) untouched — re-prompting everything is annoying.
3. The vscode test mock already stubs `showOpenDialog`/`showQuickPick`/`showInputBox`/`Uri.file`; a picker helper is unit-testable by resolving showQuickPick to `items[i]` by index (don't hard-code the label/icon strings in the test). See `keyFilePicker.test.ts`.

## 2026-06-16 - FTP feature: first user reports (issues #14/#15/#16) + latent-bug cluster

Three issues filed against the freshly-shipped 1.0.0 FTP feature, all rooted in FTP differing from SSH/SFTP. Fixing them surfaced a wider cluster of "SSH defaults leaking onto FTP" bugs.

- **#14 same-host copy threw**: FTP has no shell `cp`. The fix reuses the existing `copyFileCrossHost`/`copyFolderCrossHost` with src === dest (download then re-upload); the FTP serialization queue makes read->write on one connection safe. MUST guard "copy a folder into its own subtree" - the recursive copy re-lists the just-created dest and loops forever; SSH `cp -r` is protected by the shell, the FTP streaming path is not.
- **#15 timestamps all 1970 ("56 years ago")**: `basic-ftp` only fills `FileInfo.modifiedAt` for MLSD servers; LIST-mode servers (most) expose only `rawModifiedAt` (`"Jun 11 14:35"`, `"Mar 3 2021"`, DOS `"06-16-24 10:30AM"`). The old `modifiedAt ? getTime() : 0` collapsed everything to epoch. Parse `rawModifiedAt` (`src/connection/ftpDate.ts`). Watch the unit contract: `IRemoteFile.modifiedTime` is Unix MS - `SSHConnection.sudoListFiles` had the mirror bug (`getTime() / 1000` -> seconds -> also 1970).
- **#16 expandAll "command not found"**: `list.expandRecursively` is VS Code 1.94+ only, but `engines.vscode` is `^1.85.0`. Any built-in command gated by a newer engine than your floor will throw on older hosts. Wrap in try/catch and fall back (here: `TreeView.reveal`, since 1.40), narrowing the catch to the "not found" message so a real mid-run error does not double-expand.

LESSONS:
1. When a new connection type implements an existing interface (FTP vs SSH), the bugs cluster where callers assume the OLD type's defaults: SSH-only methods (`sudoWriteFile`, `enableSudoMode`) called without a `capabilities` guard -> TypeError; the SSH default port 22 hard-coded as `port || 22` -> FTP-on-21 host lookups silently miss (remove/rename do nothing, import duplicates). Grep the WHOLE repo for the old default (`|| 22`) and SSH-only method names after adding a protocol - one shared helper (`effectiveHostPort`) beats N inline copies.
2. Menus hide via `when` clauses, but keybindings, the command palette, and programmatic calls BYPASS them. A feature that is "hidden for FTP" in `package.json` is still reachable - guard in CODE (`ensureCapability` at the handler, `assertCapability` at the service sink), never rely on menu gating for correctness. (issue #14 was hit via Ctrl+V; the saveAsRoot crash via the palette.)
3. Don't trust a sub-agent's bug list wholesale - verify each against the real code. Of 9 "latent bugs" reported, 2 were non-bugs (symlink DELE is correct; recursive dir-delete matches SSH `rm -rf`) and 1 (handlePermissionDenied) was already guarded at every caller (kept as a cheap backstop, not billed as a live fix).
4. Inject `now` into any date parser so year-rollover / relative-time logic is deterministic under test, and reject calendar overflow explicitly (`new Date(2025, 1, 29)` silently becomes Mar 1 - round-trip the components and return undefined if they don't match).

## 2026-06-17 - FTP "550 ... operation failed" is a SERVER refusal, not our bug (issue #17)

A user on shared hosting reported three errors at once (screenshot only, no body text): "FTP delete failed: 550 Delete operation failed.", "FTP read failed: 550 Failed to open file.", "FTP delete failed: 550 Remove directory operation failed." The tree listed files fine but every file op failed.

Reproduced on docker vsftpd (`src/integration/docker-ftp-permission.test.ts`) with root-owned fixtures planted via `docker exec`: a directory chmod 755 owned by root (FTP user can LIST + enter) whose contents the FTP user cannot delete (DELE -> "550 Delete operation failed."), read (RETR of a 0600 file -> "550 Failed to open file."), or rmdir (RMD in a non-writable parent -> "550 Remove directory operation failed."). All three exact messages matched the screenshot; a file the FTP user OWNS deletes cleanly (control).

Fix is NOT functional - the server legitimately refuses (no FTP client could override, and FTP has no sudo). `describeFtpFailure(label, error)` in `FTPConnection` classifies reply code 550 (via `error.code`, the numeric code basic-ftp's FTPError sets) and appends a permission/ownership explanation while keeping the raw server text. Applied once in `runOp`, so every op (delete/read/write/rename/mkdir/list/stat) benefits.

LESSONS:
1. FTP reply-code taxonomy is the reliable signal, not the message text (which is server-specific): 530 = auth (already mapped), 550 = access/permission/not-found, 553 = filename not allowed. Classify on `error.code`, never on substring-matching the human reply. vsftpd's DELE-fail vs RMD-fail messages differ ("Delete operation failed." vs "Remove directory operation failed.") - both are 550, so code-based gating catches both.
2. "LIST works but every mutation 550s" = the account can browse but not modify (files owned by another user / non-writable parent), the classic shared-hosting layout, NOT a path bug. Prove it by checking that the SAME path lists fine and that an OWNED item deletes - if so, the server is refusing and the honest fix is a transparent, actionable message, not code that pretends to make a forbidden op succeed.
3. When the issue body is just a screenshot, READ the screenshot (download the GitHub user-attachments asset with curl, copy into the repo dir so the path is a Windows path, open with the image reader) - it carried THREE errors, not just the one in the title, which together pinned the root cause to permissions rather than any single failing command.
